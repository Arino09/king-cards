# 基于《卡牌游戏规则》的前端单机 Web 应用计划

## 现状与约束

- 仓库目前仅有规则与工程说明（[doc/卡牌游戏规则.md](doc/卡牌游戏规则.md)、[doc/工程规则.md](doc/工程规则.md)），**无现成代码**，属全新前端工程。
- 工程规则要求：**文档、注释、AI 输出使用简体中文**；功能模块交付时需在 [doc/进度文档.md](doc/进度文档.md) 追加一条汇总（若文件不存在则创建）。

---

## 1. 技术选型（仅前端）

| 层级 | 建议 | 理由 |
|------|------|------|
| 构建与语言 | **Vite + TypeScript** | 开发体验好、类型有利于规则引擎与 AI 的维护 |
| UI | **React 18** | 手牌/多面板状态多，组件化清晰 |
| 状态 | **Zustand** 或 **useReducer + Context**（二选一） | 对弈状态集中、便于调试；若希望依赖更少可用 useReducer |
| 样式 | **CSS Modules** 或 **Tailwind**（二选一） | 卡牌网格与弹层布局；若追求极简可 CSS Modules |
| 测试 | **Vitest**（规则引擎与 AI 单元测） | 规则与胜负分支多，必须有自动化回归 |

不引入后端；存档若需要可后续再加 `localStorage`，首版可按“无持久化”处理。

## 2. 交互与信息架构

### 2.1 核心对弈流程（与你的需求对齐）

- **出牌**：玩家从手牌选一张 → 确认；AI 在同期（或先后手按规则）选定 → **统一结算**。
- **比较与反馈**：调用规则引擎得到本回合结果（谁输、是否双败；功能牌开放后再展示复杂效果），**界面明确展示文案**（例如“你败，该牌进入弃牌堆”），再进入下一轮可选牌。
- **弃牌堆与可见牌组**：主界面提供固定入口（按钮或侧栏），**随时打开**查看：
  - 己方 / 对方的**弃牌堆**（列表或分组展示）；
  - 己方 / 对方的**可见牌组**（规则中一般 &lt;10 张的明牌堆）；
  - 对方**不可见牌组**对玩家仅显示**数量或背面**（不泄露具体牌），符合文档“猜测与不可预测性”。

### 2.2 开局与重开（胜败后交互）

- **不设胜利/失败专门页**：检测到终局条件后，**立即提供“开始新游戏”**（或自动跳转回开局配置），无额外剧情或统计弹窗（可按你要求极简）。
- **敌方牌组**：每局随机生成可见/不可见牌组（在约束内：含国王与平民、总数与规则一致）。
- **开局自定义**：开局页允许玩家从牌池**勾选/增减**构成己方 10 张手牌（**必须含国王与平民**，其余在规则允许范围内）；校验不通过则提示并禁止开始。

### 2.3 建议界面区块（实现时参考）


## 3. 领域模型与规则引擎（先于 UI 落地）

### 3.1 基础牌与补充基础卡列表（与 [doc/卡牌游戏规则.md](doc/卡牌游戏规则.md) 卡牌说明一致）

下列为规则文档中的 **基础卡** 与 **补充基础卡**（不含「额外功能卡」）。等级用于比大小；链条为 `1 > 1.1 > 1.5 > 2 > 2.5 > 3 > 1（仅平民对国王）> 3.5`，同等级双败，**平民 > 国王**、**其余等级 3 不能胜国王**。

| 分类 | 等级 | 名称 |
|------|------|------|
| 基础卡 | 1 | 国王 |
| 基础卡 | 2 | 大臣 |
| 基础卡 | 3 | 平民 |
| 补充基础卡 | 1.1 | 贵族 |
| 补充基础卡 | 1.5 | 护卫 |
| 补充基础卡 | 2.5 | 侍女 |
| 补充基础卡 | 3.5 | 盗贼 |
| 补充基础卡 | 1 或 3 | 弑君者 |
| 补充基础卡 | 2 或 3 | 乱党 |

**首版 `enabledCardKinds` 建议**：

- **默认开放**（仅静态等级比较、无功能牌逻辑）：国王、大臣、平民、贵族、护卫、侍女、盗贼。
- **默认关闭**（与当前计划第 6 节一致：动态规则与细则待定后再开）：**弑君者**、**乱党**。
- **额外功能卡**（刺客、伪神、占卜师、影子、命运）不在上表「基础/补充基础」之列，**首版始终关闭**，见第 6 节。

---

建议目录划分（示例）：

- `src/game/types.ts`：卡牌类型枚举、等级、实例 ID（用于“同一张牌不能连续出两次”）。
- `src/game/compare.ts`：基础等级胜负、**同阶双败**、**平民 &gt; 国王**、其余 3 不能胜国王（见文档表格与说明）。
- `src/game/special.ts`（可选/分期）：功能牌与动态牌逻辑；**复杂结算顺序可后补**，代码可先占位。
- `src/game/state.ts`：完整 `GameState`（双方手牌、出牌区、弃牌、可见/不可见牌组、上一张出的牌 ID；功能牌相关字段可在屏蔽期不参与对局）。
- `src/game/validation.ts`：出牌合法性（败者牌不可再出、不可连续同一张；功能牌条件在开放后再启用）。
- **`enabledCardKinds` / 特性开关**：集中定义**当前版本对玩家与 AI 开放的牌种**；未开放牌种不出现在组牌界面与随机敌方生成中，AI 着法枚举亦只针对开放集。

**实现顺序建议**：先实现 **3.1 中「默认开放」七类牌** 闭环并上线可玩 → 弑君者、乱党与功能牌可在仓库内**先写后关**，待第 6 条细则与结算表就绪后，改 `enabledCardKinds` 即可开放。

## 4. 敌人 AI：目标导向 + 剪枝搜索

### 4.1 信息集

- AI **完全知晓己方**手牌与不可见牌组。
- 对**玩家手牌**：仅能通过已见信息推断（可见牌组、弃牌、已打出牌）；首版可采用 **确定性启发式 + 对玩家手牌的均匀/启发式信念**（不必一上来就做完整贝叶斯）。

### 4.2 算法建议（满足“聪明”与“剪枝”）

- **着法生成**：枚举 AI 当前所有合法牌；**首版仅针对已开放牌池**（与 `enabledCardKinds` 一致），功能牌开放后再扩展条件分支。
- **搜索**：**极小极大 + Alpha-Beta 剪枝**，深度可配置（例如 2～4 回合 lookahead，视性能调整）。
- **叶节点估值 `evaluate(state)`**（启发式）：例如
  - 手牌数量差、剩余高阶牌、对方疑似威胁、功能牌剩余价值；
  - 终局直接返回极大/极小值。
- **对手节点**：若不做信念模型，可用 **“假设对手出当前最优应手”** 的乐观/悲观近似；进阶可改为 **Expectiminimax**（对玩家隐藏牌分布采样期望）。

文档要求：**AI 在能赢时不会乱出牌** —— 在估值中可对“已必胜局面”大幅提高赢面权重，并避免无意义冒险（需在 `evaluate` 与着法排序中体现）。

### 4.3 性能

- 着法排序（按 `evaluate` 粗排）可提升剪枝效率。
- 主线程卡顿则把搜索放入 `Worker`（可选，非首版必须）。

## 5. 交付顺序（建议迭代）

1. **脚手架**：Vite + React + TS + Vitest，简体中文注释与 UI 文案。
2. **规则引擎 + 测试**：完整闭环到“一方手牌耗尽”。
3. **最小 UI**：开局配置、出牌、结果条、查看弃牌/可见牌组。
4. **AI**：合法着法 + minimax + alpha-beta + 估值调参。
5. **打磨**：动画可后做；无障碍与移动端适配按需要追加。
6. **功能牌开放**（后续里程碑）：补齐第 6 条中的结算顺序表，打开 `enabledCardKinds` 并补全测试与 AI 分支。

## 6. 功能牌与复杂结算（首版不开放，可先实现后屏蔽）

以下涉及**多牌互动时的结算顺序**与**动态等级判定时机**，与 [doc/卡牌游戏规则.md](doc/卡牌游戏规则.md) 中的功能牌、弑君者、乱党等相关。**首版不对玩家开放**：不在组牌与随机敌方中出现；若代码中已编写对应分支，通过**特性开关**保持关闭，避免半套规则上线。

待后续版本专门处理时，再在代码或文档中钉死**唯一确定**的“结算顺序表”，例如：

- **占卜师**与**影子**等同回合或跨回合的交互顺序。
- **弑君者 / 乱党** 动态等级与“国王/大臣是否在场”的判定时机。
- **命运**（交换双方可见卡组）与当前回合比较的先后。
- **伪神** “最少隔 3 次” 的计数口径；**刺客**与**伪神**的优先级（文档：刺客低于伪神）。

**实施策略**：规则引擎可保留扩展点（类型定义、占位结算函数、未启用时的单元测跳过或 `@todo`）；**玩家可见玩法以基础牌池为准**，上述细则与功能牌**后续一并开放**，避免首版陷入未定规则。
